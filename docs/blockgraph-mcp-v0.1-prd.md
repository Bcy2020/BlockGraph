# BlockGraph MCP v0.1 PRD

## 1. Background And Core Idea

BlockGraph MCP is not primarily a code visualization tool or a documentation generator. It is an implementation aid for an architecture-first repository maintenance method.

The method is:

1. Maintain a mechanical code fact graph that records files, symbols, calls, imports, routes, event handlers, and other code-level facts.
2. Maintain a semantic block decomposition graph that describes business modules, responsibilities, ports, and connections.
3. Maintain business flow graphs that describe how user/system entrypoints activate blocks and code entities.
4. Keep explicit mappings between the semantic graphs and the mechanical code graph.
5. Require agents to edit the semantic model through structured MCP tools, not by hand-writing JSON.
6. Require draft graph changes to pass compilation/validation before they can be promoted into the accepted architecture graph.

The goal is to make a repository easier for coding agents to maintain:

- During initialization, an agent scans the repository, builds a first architecture model, validates it, and commits a snapshot.
- During later maintenance, an agent queries the model before reading or editing code, so it can start from relevant blocks and flows instead of scanning the entire repository from scratch.
- During feature work, bugfixes, and refactors, future versions will require the agent to update or confirm the architecture model before and after code changes.

In v0.1, only the initialization loop is required.

## 2. Product Positioning

BlockGraph MCP v0.1 is a constrained graph editor for code repositories.

It edits and validates:

- draft block graphs
- draft flow graphs
- mappings from blocks/flows to code entities

It reads and stores:

- code fact graphs generated by a scanner
- accepted architecture graphs
- immutable snapshots tied to git SHAs

The MCP server is the structured interface through which agents create, edit, validate, promote, and query these graphs.

## 3. v0.1 Goals

v0.1 must implement a working initialization loop for small TypeScript / React / Node repositories.

Required outcome:

1. Start initialization for a repository.
2. Scan the repository and generate a basic code fact graph.
3. Suggest candidate blocks from simple repository heuristics.
4. Allow an agent to create a draft block graph through MCP tools.
5. Allow an agent to attach code evidence to blocks and flow steps.
6. Compile and validate draft blocks.
7. Promote valid draft blocks into the accepted graph.
8. Compile the whole draft graph.
9. Commit an accepted snapshot tied to a git SHA.
10. Provide tests proving the loop works on a fixture repository and can run against a cloned external repository.

## 4. Non-Goals For v0.1

Do not implement these in v0.1:

- Multi-language support beyond TypeScript / JavaScript.
- Visual graph UI.
- Runtime tracing.
- Playwright-driven click tracing.
- OpenTelemetry integration.
- Neo4j or external graph databases.
- Automatic perfect module decomposition.
- Automatic business-code modification.
- Full architecture-first feature-change protocol.
- Complex refactor support.
- Distributed storage or server authentication.

These may be future versions after the initialization loop is stable.

## 5. Target Tech Stack

Required:

- Runtime: Node.js.
- Language: TypeScript.
- MCP server implementation: TypeScript.
- Storage: SQLite.
- Tests: Vitest or Node test runner.
- Package manager: choose one and document it; pnpm is preferred.

Code scanning:

- Use TypeScript Compiler API, ts-morph, or Tree-sitter.
- Choose the option with the lowest implementation risk.
- The scanner does not need to be perfect in v0.1, but it must produce stable IDs and useful evidence.

## 7. Core Concepts

### 7.1 Code Fact Graph

The code fact graph is the mechanical graph generated from the repository.

It contains:

- `CodeEntity`
- `CodeEdge`

Agents should not manually create arbitrary code facts in v0.1. Code facts come from `scan_repo`.

### 7.2 Draft Graph

The draft graph is the editable working graph.

All agent edits must first go to the draft graph:

- create block
- attach code entity
- create port
- connect ports
- create flow
- append flow step
- mark unknown boundary

The accepted graph must not be edited directly.

### 7.3 Accepted Graph

The accepted graph contains promoted blocks and flows that passed validation.

A draft block can enter the accepted graph only through `promote_draft_block`.

### 7.4 Snapshot

A snapshot is an immutable record of an accepted graph at a specific repository git SHA.

Snapshots are append-only in v0.1. Do not overwrite or delete old snapshots.

## 8. Data Model

The implementation must define TypeScript types and SQLite tables for the following models.

IDs must be stable strings. UUIDs are acceptable for graph entities. Code entity IDs should be deterministic when practical, for example based on repo-relative path, symbol kind, name, and location.

### 8.1 CodeEntity

Fields:

- `id: string`
- `type: "file" | "function" | "class" | "method" | "component" | "route" | "api_call" | "event_handler"`
- `name: string`
- `file_path: string`
- `start_line: number`
- `end_line: number`
- `metadata: object`

Implementation requirements:

- `file_path` must be repository-relative.
- `start_line` and `end_line` must be 1-based.
- A React function component can be typed as `component`.
- A callback passed to `onClick`, `onSubmit`, or `onChange` can be typed as `event_handler`.
- A `fetch`, `axios`, or obvious HTTP client call can be typed as `api_call`.

### 8.2 CodeEdge

Fields:

- `id: string`
- `type: "imports" | "calls" | "renders" | "handles_event" | "fetches"`
- `source_entity_id: string`
- `target_entity_id: string | null`
- `confidence: number`
- `evidence: Evidence[]`

Implementation requirements:

- `confidence` must be between `0` and `1`.
- `target_entity_id` may be null when the target cannot be resolved.
- Unresolved edges should still be stored when they provide useful evidence.

### 8.3 Block

Fields:

- `id: string`
- `parent_id: string | null`
- `name: string`
- `purpose: string`
- `status: "draft" | "accepted" | "stale" | "disputed"`
- `confidence: number`

Implementation requirements:

- Blocks are semantic modules.
- Blocks may be nested.
- v0.1 must support at least root blocks and one level of child blocks.
- More nesting is allowed if it does not increase implementation risk.

### 8.4 BlockCodeMapping

Fields:

- `id: string`
- `block_id: string`
- `code_entity_id: string`
- `role: string`
- `evidence: Evidence[]`

Implementation requirements:

- A block can map to many code entities.
- A code entity can map to many blocks.
- `role` examples: `owns`, `uses`, `entrypoint`, `adapter`, `helper`.

### 8.5 Port

Fields:

- `id: string`
- `block_id: string`
- `name: string`
- `direction: "in" | "out"`
- `contract: string`

Implementation requirements:

- Ports represent block boundaries.
- The contract may be natural language in v0.1.
- Later versions may add structured schemas.

### 8.6 Connector

Fields:

- `id: string`
- `source_port_id: string`
- `target_port_id: string`
- `protocol: string`
- `evidence: Evidence[]`

Implementation requirements:

- Connectors represent explained cross-block interactions.
- `protocol` examples: `function_call`, `http`, `event`, `state`, `render`, `unknown`.

### 8.7 Flow

Fields:

- `id: string`
- `name: string`
- `entrypoint_entity_id: string`
- `status: "draft" | "accepted" | "stale" | "disputed"`

Implementation requirements:

- A flow represents an entrypoint-triggered business process.
- v0.1 flows may be manually created by the agent from scanner evidence.

### 8.8 FlowStep

Fields:

- `id: string`
- `flow_id: string`
- `order: number`
- `block_id: string`
- `code_entity_id: string`
- `trigger: string`
- `evidence: Evidence[]`

Implementation requirements:

- Steps must be ordered.
- Each step must bind both a block and a code entity.
- v0.1 does not need branching flow execution.

### 8.9 UnknownBoundary

Fields:

- `id: string`
- `related_entity_ids: string[]`
- `reason: string`
- `evidence: Evidence[]`
- `status: "draft" | "accepted"`

Implementation requirements:

- Unknown boundaries are explicit acknowledgements of unresolved or unexplained cross-module interactions.
- They are allowed when the agent cannot confidently model a connector.

### 8.10 Evidence

Fields:

- `file_path: string`
- `start_line: number`
- `end_line: number`
- `code_entity_id?: string`
- `note?: string`

Implementation requirements:

- Evidence must reference repo-relative file paths.
- Line ranges must be valid for files in the scanned repository.
- Evidence is required for block-code mappings, connectors, and flow steps.

### 8.11 Snapshot

Fields:

- `id: string`
- `git_sha: string`
- `created_at: string`
- `accepted_graph_version: string`

Implementation requirements:

- Snapshots are immutable.
- `commit_snapshot` must fail if the graph has compile errors.

## 9. MCP Tools

The server must expose the following tools.

All tools must return structured JSON with:

- `ok: boolean`
- `data?: object`
- `errors?: Diagnostic[]`
- `warnings?: Diagnostic[]`

`Diagnostic` fields:

- `code: string`
- `message: string`
- `entity_id?: string`
- `severity: "error" | "warning"`
- `suggested_fix?: string`

### 9.1 begin_initialization

Input:

```json
{
  "repo_path": "string"
}
```

Behavior:

- Create or reset an initialization session for the repository.
- Initialize SQLite storage if needed.
- Store repo path and session metadata.
- Do not scan code yet.

Validation:

- `repo_path` must exist.
- `repo_path` must be a directory.

Output:

```json
{
  "session_id": "string",
  "repo_path": "string"
}
```

### 9.2 scan_repo

Input:

```json
{
  "repo_path": "string"
}
```

Behavior:

- Scan TypeScript, TSX, JavaScript, and JSX files.
- Ignore `node_modules`, build outputs, lockfiles, and hidden VCS directories.
- Generate `CodeEntity` and `CodeEdge` records.
- Persist the code fact graph.

Minimum scanner requirements:

- Identify files.
- Identify exported and local functions.
- Identify classes and methods where practical.
- Identify React function components in TSX/JSX.
- Identify JSX event handlers for `onClick`, `onSubmit`, and `onChange`.
- Identify basic import/export edges.
- Identify obvious function calls where practical.
- Identify `fetch(...)`, `axios.*(...)`, or obvious HTTP client calls as `api_call`.

Output:

```json
{
  "entity_count": 0,
  "edge_count": 0,
  "unsupported_file_count": 0
}
```

### 9.3 list_code_entities

Input:

```json
{
  "filter": {
    "type": "string?",
    "file_path": "string?",
    "name_contains": "string?"
  }
}
```

Behavior:

- Return matching code entities.
- Support empty filter.

### 9.4 list_code_edges

Input:

```json
{
  "filter": {
    "type": "string?",
    "source_entity_id": "string?",
    "target_entity_id": "string?"
  }
}
```

Behavior:

- Return matching code edges.
- Support empty filter.

### 9.5 suggest_block_candidates

Input:

```json
{
  "strategy": "directory | route | component | mixed"
}
```

Behavior:

- Suggest candidate blocks from the code graph.
- v0.1 can use simple heuristics.

Minimum strategies:

- `directory`: group entities by top-level or major source directory.
- `component`: group React components with nearby event handlers and imports.
- `mixed`: combine directory and component heuristics.

Output:

```json
{
  "candidates": [
    {
      "name": "string",
      "reason": "string",
      "code_entity_ids": ["string"],
      "confidence": 0.0
    }
  ]
}
```

### 9.6 create_block

Input:

```json
{
  "parent_id": "string|null",
  "name": "string",
  "purpose": "string"
}
```

Behavior:

- Create a draft block.
- If `parent_id` is provided, it must reference an existing block.
- Accepted graph must not be directly edited.

Output:

```json
{
  "block_id": "string",
  "status": "draft"
}
```

### 9.7 attach_code_entity

Input:

```json
{
  "block_id": "string",
  "code_entity_id": "string",
  "role": "string",
  "evidence": [
    {
      "file_path": "string",
      "start_line": 1,
      "end_line": 1,
      "code_entity_id": "string",
      "note": "string"
    }
  ]
}
```

Behavior:

- Attach a code entity to a draft or accepted block.
- For initialization, normally attach to draft blocks before promotion.

Validation:

- Block must exist.
- Code entity must exist.
- Evidence must reference valid file paths and line ranges.

### 9.8 create_port

Input:

```json
{
  "block_id": "string",
  "name": "string",
  "direction": "in | out",
  "contract": "string"
}
```

Behavior:

- Create a draft port for a block.

Validation:

- Block must exist.
- Direction must be valid.

### 9.9 connect_ports

Input:

```json
{
  "source_port_id": "string",
  "target_port_id": "string",
  "protocol": "string",
  "evidence": [
    {
      "file_path": "string",
      "start_line": 1,
      "end_line": 1,
      "code_entity_id": "string",
      "note": "string"
    }
  ]
}
```

Behavior:

- Create a draft connector between two ports.

Validation:

- Both ports must exist.
- Source port should have direction `out`.
- Target port should have direction `in`.
- Evidence must be valid.

### 9.10 create_flow

Input:

```json
{
  "name": "string",
  "entrypoint_entity_id": "string"
}
```

Behavior:

- Create a draft flow.

Validation:

- Entrypoint code entity must exist.
- Entrypoint should usually be an `event_handler`, `route`, `component`, or `function`.

### 9.11 append_flow_step

Input:

```json
{
  "flow_id": "string",
  "block_id": "string",
  "code_entity_id": "string",
  "trigger": "string",
  "evidence": [
    {
      "file_path": "string",
      "start_line": 1,
      "end_line": 1,
      "code_entity_id": "string",
      "note": "string"
    }
  ]
}
```

Behavior:

- Append a step to a draft flow.
- Assign the next integer `order`.

Validation:

- Flow must exist.
- Block must exist.
- Code entity must exist.
- Evidence must be valid.

### 9.12 mark_unknown_boundary

Input:

```json
{
  "related_entity_ids": ["string"],
  "reason": "string",
  "evidence": [
    {
      "file_path": "string",
      "start_line": 1,
      "end_line": 1,
      "code_entity_id": "string",
      "note": "string"
    }
  ]
}
```

Behavior:

- Record an unresolved or intentionally unspecified boundary.
- This is used when a cross-block edge cannot be confidently modeled as a connector.

Validation:

- Related code entities must exist.
- Reason must be non-empty.
- Evidence must be valid.

### 9.13 compile_draft_block

Input:

```json
{
  "block_id": "string"
}
```

Behavior:

- Validate a single draft block and its directly associated mappings, ports, connectors, and flow steps.
- Return diagnostics.
- Do not promote automatically.

Required error checks:

- Block exists.
- Block has non-empty `name`.
- Block has non-empty `purpose`.
- Block has at least one attached code entity, unless it is a root block.
- Attached code entities exist.
- Evidence file paths and line ranges are valid.
- Ports belong to existing blocks.
- Connectors referencing this block's ports point to existing ports.
- Flow steps referencing this block bind existing code entities.

Required warning checks:

- Cross-block code edges involving this block exist but have no connector and no unknown boundary.
- Block confidence is low.

Output:

```json
{
  "block_id": "string",
  "can_promote": true,
  "errors": [],
  "warnings": []
}
```

### 9.14 promote_draft_block

Input:

```json
{
  "block_id": "string"
}
```

Behavior:

- Run `compile_draft_block`.
- If there are no errors, set block status to `accepted`.
- Promote directly associated ports, mappings, connectors, and flow steps when safe.

Validation:

- Must fail if `compile_draft_block` returns any error.

### 9.15 compile_draft_graph

Input:

```json
{}
```

Behavior:

- Validate the entire draft/accepted graph before snapshot commit.

Required error checks:

- All non-root accepted/draft blocks must have evidence mappings.
- All accepted connectors must reference existing ports.
- All accepted flow steps must reference existing flows, blocks, and code entities.
- Evidence must be valid.
- No accepted entity may reference a deleted or missing code entity.

Required warning checks:

- Draft blocks remain unpromoted.
- Cross-block code edges exist without connector or unknown boundary.
- No flows exist.

Output:

```json
{
  "can_commit": true,
  "errors": [],
  "warnings": []
}
```

### 9.16 commit_snapshot

Input:

```json
{
  "git_sha": "string"
}
```

Behavior:

- Run `compile_draft_graph`.
- If there are no errors, create an immutable snapshot of the accepted graph.
- Store the snapshot with the git SHA and timestamp.

Validation:

- Must fail if compile has errors.
- Must fail if `git_sha` is empty.

### 9.17 query_block

Input:

```json
{
  "block_id": "string"
}
```

Behavior:

- Return block details, ports, mappings, connectors, and related flow steps.

### 9.18 query_symbols_by_block

Input:

```json
{
  "block_id": "string"
}
```

Behavior:

- Return all code entities mapped to the block.

## 11. Initialization Agent Workflow

The initialization agent must follow this exact workflow:

1. Call `begin_initialization({ repo_path })`.
2. Call `scan_repo({ repo_path })`.
3. Call `list_code_entities({})` and inspect the result.
4. Call `suggest_block_candidates({ strategy: "mixed" })`.
5. Create a root block using `create_block`.
6. Create first-level business blocks using `create_block`.
7. Attach code entities to each business block using `attach_code_entity`.
8. Create ports for meaningful block boundaries using `create_port`.
9. Connect ports using `connect_ports` when evidence exists.
10. Create at least one flow using `create_flow`.
11. Append at least two flow steps using `append_flow_step`.
12. For each business block, call `compile_draft_block`.
13. If a block has errors, modify the draft graph and compile again.
14. When a block has no errors, call `promote_draft_block`.
15. Call `compile_draft_graph`.
16. If the graph has errors, modify the draft graph and compile again.
17. Call `commit_snapshot({ git_sha })`.

Agents must not write graph JSON directly.

## 12. Validation Philosophy

The compiler/validator should not pretend the architecture model is perfectly complete.

It should enforce:

- valid references
- valid evidence
- block-code mappings
- flow-code mappings
- explicit treatment of unknown boundaries

It should allow:

- warnings for incomplete but useful models
- unknown boundaries with reasons and evidence
- low-confidence mappings

It should reject:

- accepted blocks with no code evidence, except root blocks
- missing code entity references
- invalid line evidence
- invalid connector endpoints
- invalid flow step references
- snapshot commits when graph compile has errors

## 13. Testing Requirements

### 13.1 Unit Tests

Must test:

- Create a block.
- Create a child block.
- Attach a valid code entity.
- Reject attaching a missing code entity.
- Create ports.
- Connect ports.
- Reject connector with missing port.
- Create flow.
- Append flow step.
- Reject flow step with missing block or code entity.
- Reject non-root block compile when it has no code entity mapping.
- Allow root block compile with no code entity mapping.
- Promote only after successful compile.
- Reject snapshot commit when compile has errors.

### 13.2 Scanner Fixture Test

Create a fixture repository under `fixtures/ts-react-auth`.

Minimum files:

```text
fixtures/ts-react-auth
├── package.json
├── src
│   ├── LoginForm.tsx
│   ├── authService.ts
│   ├── apiClient.ts
```

Fixture behavior:

Scanner test must verify:

- Event handler is detected.
- Service function is detected.
- API call is detected.

### 13.3 Initialization Flow Test

Using `fixtures/ts-react-auth`, test a scripted initialization:

1. Begin initialization.
2. Scan repo.
3. Create root block: `Auth Feature`.
4. Create blocks:
   - `Auth UI`
   - `Auth Service`
   - `API Client`
5. Attach code entities to blocks.
6. Create ports:
   - `Auth UI.out.submitCredentials`
   - `Auth Service.in.loginRequest`
   - `Auth Service.out.httpAuthRequest`
   - `API Client.in.request`
7. Connect ports with evidence.
8. Create flow: `Submit Login`.
9. Add flow steps:
   - Login form submit handler
   - auth service login function
   - API client request
10. Compile each block.

13. Commit snapshot.

Test must assert:

- Snapshot exists.
- Blocks are accepted.
- Flow exists.
- Flow has ordered steps.
- Mappings exist between blocks and code entities.

### 13.4 External Repository Initialization Test

Provide a script:

```bash
pnpm test:external-init --repo <repo_url> --ref <commit_sha>
```

Behavior:

- Clone the target repository to a temporary directory.
- Checkout the fixed ref.
- Run `scan_repo`.
- Run `suggest_block_candidates`.
- Perform a minimal scripted initialization using candidate blocks.
- Compile graph.
- Commit snapshot if there are no errors.

The external test may allow warnings.

The external test must not require perfect module semantics. It only proves that the initialization pipeline can run against a real repository.

Recommended target:

- Use a small public TypeScript/React repository with a fixed commit SHA.
- The chosen repository and SHA must be documented in the test script or README.

- The test runs against one documented public repository and fixed commit SHA.
- Warnings are allowed.
- Errors fail the test.

## 17. Acceptance Criteria

The v0.1 implementation is accepted only if all conditions below are met:

1. MCP tools listed in section 9 are implemented.
2. Storage persists code graph, draft graph, accepted graph, flows, evidence, unknown boundaries, and snapshots.
3. Scanner works on the fixture repository.
4. Draft graph edits do not directly mutate accepted graph except through promotion.
5. `compile_draft_block` prevents promotion of invalid non-root blocks.
6. `compile_draft_graph` prevents snapshot commit when errors exist.
7. Fixture initialization flow test passes end to end.
8. External repository smoke test script exists and is documented.
9. README explains how to run tests and start the MCP server.
10. Agent initialization guide exists.
