# BlockGraph MCP — Project Rules

## BlockGraph Mental Model

BlockGraph MCP is **not** a CRUD backend, not a Mermaid generator, not a plain call graph visualizer.

It is a **constrained graph editor** for architecture-first repository maintenance, built on three graph layers:

### Three Graph Layers

1. **Code Fact Graph** — the mechanical truth layer.
   Generated by scanner from repository source code. Contains `CodeEntity` (files, functions, classes, components, routes, event handlers, api calls) and `CodeEdge` (imports, calls, renders, handles_event, fetches). Agents do NOT manually create code facts in v0.1; they come from `scan_repo`.

2. **Block Graph** — the semantic projection layer.
   Human/agent-authored decomposition of the repository into business modules (`Block`), their boundaries (`Port`), cross-boundary interactions (`Connector`), and mappings from blocks to code entities (`BlockCodeMapping`). Blocks have purpose, confidence, and status (draft/accepted/stale/disputed).

3. **Flow Graph** — the entrypoint-triggered process layer.
   Describes how a user or system entrypoint activates a sequence of blocks and code entities (`Flow` + `FlowStep`). Each step binds both a block and a code entity with evidence.

### The State Machine

All semantic graph mutations follow this protocol:

```
draft ──compile──> valid ──promote──> accepted ──commit──> snapshot (immutable)
```

- **draft**: editable working state. All agent edits go here first.
- **compile**: validation pass. Checks references, evidence, mappings. Produces errors and warnings.
- **promote**: moves a valid draft block (and its ports, mappings, connectors, flow steps) into the accepted graph.
- **snapshot**: immutable record of the accepted graph at a specific git SHA. Append-only.

**The accepted graph MUST NOT be edited directly.** Only `promote_draft_block` can move draft entities into accepted. Only `commit_snapshot` can freeze the accepted graph.

### What This Means For Implementation

- MCP tools are the **primary product interface**. CLI is for dev/debug only.
- Agents **never write graph JSON directly**. They call MCP tools that enforce the draft/compile/promote protocol.
- The compiler is not a type checker. It enforces: valid references, valid evidence, block-code mappings, flow-code mappings, explicit unknown boundaries. It allows warnings for incomplete-but-useful models.
- Evidence is mandatory for non-root block mappings, connectors, and flow steps. Evidence must reference real file paths and valid line ranges.

## v0.1 Goals

Build a working **initialization loop** for small TypeScript / React / Node repositories:

1. Scan repository → generate code fact graph
2. Suggest candidate blocks from heuristics
3. Agent creates draft block graph through MCP tools
4. Agent attaches code evidence to blocks and flow steps
5. Compile and validate draft blocks
6. Promote valid draft blocks into accepted graph
7. Compile whole draft graph
8. Commit accepted snapshot tied to git SHA
9. Tests proving the loop works on fixture repo and can run against external repo

## v0.1 Non-Goals

Do NOT implement:

- Multi-language support (beyond TS/JS)
- Visual graph UI
- Runtime tracing / Playwright click tracing / OpenTelemetry
- Neo4j or external graph databases
- Automatic perfect module decomposition
- Automatic business-code modification
- Full architecture-first feature-change protocol
- Complex refactor support
- Distributed storage or server authentication

## Tech Stack Decisions

- **Package manager**: pnpm
- **Runtime**: Node.js
- **Language**: TypeScript
- **Test framework**: Vitest
- **Scanner**: ts-morph (lowest implementation risk; high-level API over TS Compiler API)
- **SQLite driver**: better-sqlite3 (synchronous API, fits MCP server architecture)
- **Storage**: SQLite (local file, predictable location)

## Implementation Phases

Strictly sequential. Each phase must pass validation before the next begins.

| Phase | Deliverable | Validation |
|-------|-------------|------------|
| 1 | Project skeleton, SQLite store, data model types, CRUD service | `pnpm test` passes for graph CRUD |
| 2 | MCP server skeleton, draft graph editor tools | Tool handlers testable without MCP client |
| 3 | Scanner (`scan_repo`, `list_code_entities/edges`), fixture repo | Scanner fixture test passes |
| 4 | Compiler, promotion, snapshot (`compile_draft_block`, `promote_draft_block`, `compile_draft_graph`, `commit_snapshot`) | Unit tests for errors/warnings, snapshot blocked on compile errors |
| 5 | Initialization flow test over fixture, README, agent guide | Full init test passes, snapshot has accepted blocks + flow steps |
| 6 | External repository smoke test (`pnpm test:external-init`) | Runs against one documented public repo at fixed SHA |

## Testing Rules

- Every MCP tool must have unit tests covering success and failure paths.
- `compile_draft_block` must reject non-root blocks with no code entity mapping.
- `promote_draft_block` must fail if compile has errors.
- `commit_snapshot` must fail if `compile_draft_graph` has errors.
- Evidence must reference real file paths and valid 1-based line ranges.
- Fixture repo (`fixtures/ts-react-auth`) must have: LoginForm.tsx, authService.ts, apiClient.ts, routes/auth.ts.

## Tool Constraints

- **Fetch**: `WebFetch` is broken. Always use `mcp__fetch__fetch` for URL fetching.
